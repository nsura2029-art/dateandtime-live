# Wikimedia "On This Day" structured API — full field report

**Verdict up front:** the Wikimedia Feed API `onthisday` family is the single best free source for building an on-this-day database: structured JSON, 5 item types (selected/events/births/deaths/holidays), 14 Wikipedia languages, no API key strictly required (a good `User-Agent` is enough at low volume), ~200–1,500 items per day depending on date, CC BY-SA licensed. Every claim below was verified with **live requests on 2026-07-19** (all example responses are real, truncated where marked). Note: the old docs URL `https://api.wikimedia.org/wiki/Feed_API` now redirects to the consolidated catalog at `https://www.mediawiki.org/wiki/Wikimedia_APIs`; the service code is open source (`mediawiki/services/wikifeeds`).[^1^][^2^]

---

## Endpoint reference

Base URL (canonical, WMF API portal): `https://api.wikimedia.org/feed/v1/wikipedia/{lang}/`
No-auth mirror (legacy RESTBase, still live as of 2026-07-19): `https://{lang}.wikipedia.org/api/rest_v1/` — same payload, same schema, no key needed.[^3^]

| Endpoint | What it returns | Verified live? |
|---|---|---|
| `onthisday/all/{MM}/{DD}` | One JSON with all 5 lists: `selected`, `events`, `births`, `deaths`, `holidays` | ✅ `…/wikipedia/en/onthisday/all/07/19` (200)[^4^] |
| `onthisday/selected/{MM}/{DD}` | `{"selected":[…]}` — editor-curated highlights (from `Wikipedia:Selected anniversaries/<Month Day>`) | ✅ 20 items on 07/19[^5^] |
| `onthisday/events/{MM}/{DD}` | `{"events":[…]}` — all events from the day article, **sorted year-descending** | ✅ via rest_v1 mirror[^3^] |
| `onthisday/births/{MM}/{DD}` | `{"births":[…]}` — sorted year-descending | ✅ (list observed in `all` and per-type route)[^4^] |
| `onthisday/deaths/{MM}/{DD}` | `{"deaths":[…]}` — sorted year-descending (starts with 2024 deaths on 07/19) | ✅[^6^] |
| `onthisday/holidays/{MM}/{DD}` | `{"holidays":[…]}` — holidays/observances/feast days (**no `year` field**) | ✅ 11 items on 07/19[^7^] |
| `feed/availability` (rest_v1 path) | JSON matrix: which feed features exist per language | ✅[^8^] |
| `featured/{yyyy}/{mm}/{dd}` | Daily featured bundle: TFA, most-read, POTD, news, DYK (see Bonus feeds) | documented, same path family[^2^] |
| `feed/announcements` | Fundraising/notice banners | documented[^2^] |

Rules & facts:
- **MM and DD must be zero-padded 2-digit** (`07/19` works; `7/19` rejected by routing).[^2^]
- **Languages with `on_this_day` support (measured via `feed/availability`): 14 Wikipedias** — `en, de, fr, sv, pt, ru, es, ar, bs, uk, it, tr, zh, cs`.[^8^]
- The feed is **generated by the `wikifeeds` Node.js service** (open source, Gerrit mirror on GitHub) by parsing the Wikipedia *day articles* (e.g., "July 19") and the "Selected anniversaries" project pages, then hydrating every linked article with a page-summary object.[^2^][^9^]
- **Recency filter (observed, undocumented):** `events` excludes the most recent ~2 years. On 07/19 the day article contains 2024 (×2) and 2025 (×1) events, but feed `events` starts at 2018. `selected`, `deaths`, `births` are **not** filtered (feed deaths start with 2024 deaths; selected includes a 2024 item).[^3^][^5^][^6^]
- Items inside each list are **sorted by year descending** (most recent first).[^3^][^6^]

## Live-tested JSON schema (real sample, truncated)

Top-level item (identical for `selected` / `events` / `births` / `deaths`):

```jsonc
{
  "text": "Neary's, a New York City Irish pub popular with political elites, closed.",
  "pages": [ /* 1..N hydrated article objects (see below) */ ],
  "year": 2024            // integer; ABSENT on holidays
}
```

Holiday item — note `pages` can be an **empty array** (real: `"Palace Day"` on 07/19):[^7^]

```jsonc
{ "text": "Palace Day", "pages": [] }
```

Feast-day sub-bullets are flattened into individual holidays with a prefix and a literal newline:
`"text": "Christian feast day:\nAurea of Córdoba"`.[^7^]

`pages[]` object (real fields, one real truncated example — page "Neary's" from `selected/07/19`):[^5^]

```jsonc
{
  "type": "standard",
  "title": "Neary's",
  "displaytitle": "<span lang=\"en\" dir=\"ltr\"><span class=\"mw-page-title-main\">Neary&#039;s</span></span>",
  "namespace": { "id": 0, "text": "" },
  "wikibase_item": "Q128811934",
  "titles": { "canonical": "Neary's", "normalized": "Neary's",
              "display": "<span …>Neary&#039;s</span>" },
  "pageid": 77445945,
  "thumbnail":     { "source": "https://upload.wikimedia.org/wikipedia/commons/thumb/f/f3/Neary%27s_NYC.jpg/330px-Neary%27s_NYC.jpg",
                     "width": 330, "height": 248 },          // optional
  "originalimage": { "source": "https://upload.wikimedia.org/wikipedia/commons/f/f3/Neary%27s_NYC.jpg",
                     "width": 3057, "height": 2293 },        // optional
  "lang": "en", "dir": "ltr",
  "revision": "1363601931",
  "tid": "9b0232ba-7cea-11f1-a343-0c22fea9ff75",
  "timestamp": "2026-07-11T05:37:29Z",                      // page revision time
  "description": "Irish pub in New York City, 1967 to 2024", // optional (may be "")
  "description_source": "local",
  "coordinates": { "lat": 40.7583, "lon": -73.9635 },        // optional
  "content_urls": {
    "desktop": { "page": "https://en.wikipedia.org/wiki/Neary's",
                 "revisions": "https://en.wikipedia.org/wiki/Neary's?action=history",
                 "edit": "https://en.wikipedia.org/wiki/Neary's?action=edit",
                 "talk": "https://en.wikipedia.org/wiki/Talk:Neary's" },
    "mobile":  { "page": "…", "revisions": "…Special:History/…", "edit": "…", "talk": "…" }
  },
  "extract": "Neary's was an Irish pub on East 57th Street … until its closure in 2024.",
  "extract_html": "<p><b>Neary's</b> was an Irish pub … </p>",
  "normalizedtitle": "Neary's"
}
```

Schema notes for DB design (all observed):
- `thumbnail`/`originalimage` **absent on some pages**; always null-check.
- Most thumbnails are Wikimedia Commons files, but **some are enwiki-hosted fair-use images** (`upload.wikimedia.org/wikipedia/en/…`, e.g. book covers, a 1976 newspaper front page seen in 07/19 selected) — different licensing, see Licensing section.[^5^]
- `extract` is plain text (first-paragraph summary); `extract_html` keeps `<b>/<i>` markup.
- One event links multiple pages (mean ≈2–5 for events; births/deaths usually 1–2).
- `year` is a signed-ish integer; ancient entries go back to e.g. AD 64 (07/19 events) and even BC years exist on some dates (wikitext uses "484 BC" style; feed emits negative or small ints — validate before storing as unsigned).

Minimal real two-item fragments:
- event: `{"text":"The Knesset passes the controversial Nationality Bill…","pages":[…],"year":2018}` (first item of `events/07/19`)[^3^]
- holiday: `{"text":"Martyrs' Day (Myanmar)","pages":[{…"title":"Martyrs'_Day_(Myanmar)"…},{…"Myanmar"…}]}`[^7^]

## Data volume per day (measured)

Method: per-type feeds fetched live for **July 19**; large lists counted exactly from the feed's source — the "July 19" day-article wikitext sections (feed = those lists + recency filter on events). January 1 estimated from exact section byte offsets × July-19-measured bytes/entry ratios (transparent estimate).

**July 19 (fully measured):**

| List | Count | Basis |
|---|---|---|
| `selected` | **20** | exact, from feed[^5^] |
| `events` | **60 in feed** (63 in article; 2024×2 + 2025×1 filtered out) | article sections 11+13+39; feed starts 2018[^3^] |
| `births` | **228** (4 pre-1600 + 46 from 1601–1900 + 178 from 1901–present) | exact, article sections |
| `deaths` | **119** (15 + 12 + 92) | exact, article sections; feed starts with 2024 deaths[^6^] |
| `holidays` | **11** (Palace Day, 8 "Christian feast day:" entries, Martyrs' Day MM, Sandinista Day NI) | exact, from feed[^7^] |
| **Total items for one day** | **≈438** | |

**January 1 (the biggest day article: 210,913 B wikitext vs 68,949 B for July 19):**[^10^]

| List | Estimated count | Basis (section bytes ÷ measured bytes/entry) |
|---|---|---|
| events | ≈170–205 | 55,315 B events ÷ ~216–347 B/entry |
| births | ≈630–700 | 96,752 B births ÷ ~133–145 B/entry |
| deaths | ≈380–465 | 53,619 B deaths ÷ ~109–144 B/entry |
| holidays | ≈25–40 | 4,661 B holidays section |
| **Total** | **≈1,200–1,400 items** | |

So: **expect 400–450 items/day on a typical date (enwiki), up to ~1.4k on Jan 1; ~170k–200k items/year for one language.** Births are the largest category (~50%), then deaths (~27%), events (~14%), selected (~5%), holidays (~2–3%). English is by far the largest; the 13 other languages are smaller and structured differently (e.g., dewiki day pages have different section conventions — the feed normalizes this for you).

**Curation:** `selected` = human-curated "Selected anniversaries" list (the main-page quality tier, 20 items on 07/19 — the main page rotates 4–6 of them); `events` = the complete day-article list. For a quality-focused product, `selected`+`holidays` is the high-signal subset.

## Auth & rate limits

**Auth — verified:**
- `api.wikimedia.org/feed/v1/…` works **with no Authorization header** (tested anonymously → 200). An OAuth 2.0 app (client credentials / "personal API key" from the API portal) raises your limits.[^4^]
- The rest_v1 mirror (`en.wikipedia.org/api/rest_v1/feed/onthisday/…`) needs **no auth at all** — tested live, same JSON.[^3^]
- **A descriptive `User-Agent` (or `Api-User-Agent`) header with contact info is mandatory policy** for all Wikimedia APIs; generic/missing UAs are now automatically throttled/blocked (WMF engineers publicly confirmed 429 enforcement rollouts in 2025–2026). Send e.g. `User-Agent: YourApp/1.0 (https://yoursite.example/; you@example.com)`.[^11^][^12^][^13^]
- Official curl pattern:
  `curl -A "YourApp/1.0 (you@example.com)" "https://api.wikimedia.org/feed/v1/wikipedia/en/onthisday/all/07/19"`

**Rate limits (official "Rate limits" page, Nov 2024, as quoted in community references):**[^12^]
- Anonymous (per IP): **500 requests/hour**
- Personal API token: **5,000 requests/hour**
- OAuth 2.0 tokens: **5,000 requests/hour**
- Global etiquette ceiling: **≤200 requests/second**, honor `429` + `Retry-After`, back off on 503.[^11^][^14^]
- Feed responses are CDN/RESTBase-cached (regenerated when the source day-page is edited — the 07/19 deaths already contain 2024 entries, so freshness is effectively "minutes after an editor updates the day page").

## Licensing & attribution requirements

- Wikipedia text (event blurbs, extracts) is **CC BY-SA 4.0** (previously 3.0) + GFDL; the REST API terms state content is CC BY-SA 3.0/GFDL and you agree to WMF Terms of Use by using the API.[^11^][^15^]
- **Attribution requirement (CC BY-SA):** credit + license link + indication of changes. Recommended string per item or per page/screen:
  > "Text from Wikipedia contributors via the Wikimedia Feed API, licensed [CC BY-SA 4.0](https://creativecommons.org/licenses/by-sa/4.0/)."
  Each `pages[]` object gives you ready-made attribution links (`content_urls.desktop.page` and `…revisions` = history page for author credit). WMF reuse guidance: `Wikipedia:Reusing Wikipedia content`.[^15^]
- **Share-alike:** if you adapt the *text* (e.g., rewrite/translate), derivative text must be under CC BY-SA too. Storing facts (year + "X happened") as your own short factual records is not an adaptation of expression — but verbatim `extract`/`text` storage in your DB is a CC BY-SA copy; plan the DB license accordingly.
- **Images are NOT covered by the text license.** Each thumbnail is licensed individually (mostly CC BY / CC BY-SA / PD on Wikimedia Commons — check the file description page derived from the file name; some are **fair-use enwiki images**, identifiable by the `/wikipedia/en/` path prefix — do not hotlink/redistribute those). Correct approach: store the image URL + fetch license/author from Commons' `extmetadata` (`action=query&prop=imageinfo&iiprop=extmetadata`) per file, attribute the photographer.[^5^][^16^]
- Wikimedia Enterprise's marketing reiterates: ">99.9% of data … under a Creative Commons license" and "Every request has metadata clearly explaining the attached license" — the free feed carries the same obligations.[^16^]

## Bulk-crawl strategy to build your own DB

1. **Volume math:** 366 days × 6 routes = 2,196 req/language, but `onthisday/all/{MM}/{DD}` collapses it to **366 requests per language** (14 languages → 5,124 requests for everything). Measured payload ≈ **0.3–0.6 MB per `/all` day** → ~150–200 MB for all-enwiki, ~1–2 GB for all 14 languages. Trivial.
2. **Pacing:** anonymous 500 req/h → one language in ~45 min; free personal token (5,000 req/h) → **all 14 languages in ~1 hour**. Well inside limits; still add `maxlag`-style courtesy: serial requests, ≤5 req/s, exponential backoff on 429/503, honor `Retry-After`.[^12^][^14^]
3. **Caching/refresh:** day pages are edited year-round (recent deaths added daily; the events recency filter rolls forward). Strategy: full crawl once, then **re-fetch each date ~weekly**, plus a rolling re-fetch of "today/tomorrow" daily (that's when edits cluster). Store `revision`/`timestamp` fields per page to detect changes cheaply.
4. **DB shape:** `items(item_id PK, lang, mmdd, type[selected|events|births|deaths|holidays], year NULL, text, position)` + `pages(pageid PK, lang, title, wikibase_item, description, extract, thumbnail_url, thumb_w, thumb_h, coords, revision, timestamp)` + join table `item_pages(item_id, pageid, position)`. Keep raw JSON per (lang, mmdd) as a blob for re-processing — it is your replayable snapshot.
5. **Gotchas:** zero-pad MM/DD; `year` absent for holidays; `pages` may be `[]`; thumbnails optional; fair-use image paths; `extract_html` needs sanitizing if rendered; some languages' holiday names contain `\n`; Feb 29 exists (leap day list); dedupe pages across types (a person can be in both births and deaths of the same date).
6. **Fallback/secondary source:** parse day articles directly (next section) — also the only way to get languages outside the 14 supported.

## Alternative access: parsing Wikipedia day articles via the Action API

Verified working pattern (used for the measurements above):

```
GET https://en.wikipedia.org/w/api.php?action=parse&page=July_19&prop=sections&format=json
GET https://en.wikipedia.org/w/index.php?title=July_19&action=raw&section={N}
```
- `prop=sections` (new name: `prop=tocdata`) returns section index + **byte offsets** — lets you fetch Events/Births/Deaths/Holidays subsections individually (`Pre-1600`, `1601–1900`, `1901–present` on enwiki).[^10^]
- Each wikitext line `* [[YEAR]] – [[Page]], description …` = one item; nested `**` bullets under "Christian feast day" etc.

**Pros vs Feed API:** works for *any* of 300+ language editions; gives you the freshest data (no feed cache); exact counts; no hydration so far smaller payloads (day wikitext 40–210 KB vs 0.3–0.6 MB hydrated feed); usable as an independent audit of the feed.
**Cons vs Feed API:** you parse wikitext yourself (fragile templates: `{{ndash}}`, `{{ubl}}`, "AD 64", "484 BC", circa years, reference tags); you must hydrate pages yourself (extra API calls for thumbnails/extracts — that's 300–800 summary calls per day if you want parity); section structure differs per language (enwiki splits at 1600/1901, others don't); no `selected` curation (that lives on separate `Wikipedia:Selected anniversaries/<day>` pages, parseable but another format).

**Recommendation:** build the DB from the Feed API; use `action=parse`/`action=raw` only for unsupported languages or change-auditing.

## Bonus feeds (featured/picture/news/most-read)

Same `feed/v1` family (verified via live `feed/availability` + service README):[^2^][^8^]

| Feed | Endpoint pattern | Language support (measured) |
|---|---|---|
| Today's featured article | `featured/{yyyy}/{mm}/{dd}` (`tfa` field) | ~15 wikipedias (en, de, fr, sv, pt, ru, es, ar, bs, uk, it, tr, zh, cs subset list) |
| Picture of the day | `featured/{yyyy}/{mm}/{dd}` (`image` field) | **all languages** (Commons POTD) |
| Most read articles | `featured/{yyyy}/{mm}/{dd}` (`mostread`, also standalone `page/most-read/{yyyy}/{mm}/{dd}`) | all (from Pageviews API) |
| In the news | `featured/{yyyy}/{mm}/{dd}` (`news`, standalone `page/news`) | subset (en, de, fr, sv, pt, ru, es, ar, bs, uk, it, tr, zh, cs, …) |
| Did you know | `featured/{yyyy}/{mm}/{dd}` (`dyk`) | subset |
| On this day | `featured/{yyyy}/{mm}/{dd}` (`onthisday` = the *selected* list only) | the 14 above |
| Feed availability matrix | `feed/availability` | — |

A single `featured/{yyyy}/{mm}/{dd}` call per day therefore powers a whole "Today" page: TFA + POTD + most-read top-50 + news + DYK + on-this-day highlights. Payload is large (most-read embeds 50 hydrated articles) — cache daily at ~00:00 UTC+lag (feeds update shortly after UTC midnight; the page's `timestamp` fields show same-day revisions).

## Takeaways

1. **This is the data source.** Free, keyless (with UA), structured, multilingual (14), cached by WMF's CDN, and stable since 2017 (it powers the official Wikipedia apps' Explore feed).
2. Use `onthisday/all/{MM}/{DD}` — 366 calls/language; everything else is a filtered view of the same data.
3. Schema is clean but nullable: `year` missing on holidays, `pages` can be empty, thumbnails optional, some images fair-use — handle all four or your importer will crash.
4. Volume: ~440 items/day typical enwiki, ~1.4k on Jan 1 → ~170k+/year/language. Births ≈ half of all items.
5. `selected` (≈20/day) is the editor-curated tier; `events` silently drops the last ~2 years — if you want latest-year events, parse day articles via `action=parse`.
6. Rate limits are generous (500/h anon, 5k/h free token) but **User-Agent policy is now machine-enforced** — a contact-bearing UA is the single most important header.[^12^][^13^]
7. Licensing is workable: CC BY-SA text attribution (ready-made URLs in each `pages[]` object) + per-image Commons attribution; don't hotlink `/wikipedia/en/` fair-use thumbnails.
8. The legacy `rest_v1` mirror still works with zero auth but is legacy — build against `api.wikimedia.org/feed/v1/` and keep the mirror as a fallback.
9. For a full "Today" page, add one `featured/{yyyy}/{mm}/{dd}` call/day (TFA, POTD, most-read, news, DYK) — same pipeline, same license handling.
10. Keep raw per-day JSON blobs; re-crawl each date weekly (day pages are living documents — deaths from *this week* appear within hours of an editor's edit).
